# 本地保存可靠性与独占编辑设计

## 大白话摘要

现在的编辑器几乎每输入一个字，就会把整个作品库重新写入浏览器。写入失败时，大部分页面也不会明确告诉用户；同一个作品如果在两个标签页里打开，还可能互相覆盖。

这一阶段会做五件事：

1. 正文连续输入时先暂存在当前页面，停顿约 600ms 后合并保存；持续输入时最迟每 3 秒保存一次。
2. 同一个作品只允许一个标签页编辑，其他标签页只能等待、返回或在旧锁过期后接管。
3. 删除章节、批量保存选项、创建或删除手机模块卡片等操作一次写完整，不留下半成品。
4. 页面明确显示“未保存、正在保存、已保存、保存失败、编辑权丢失”。确认没有写入的失败可以重试；写入结果无法确认时先重新检查。两种情况都保留紧急备份出口。
5. 切换页面、关闭编辑器和恢复完整备份之前，先处理尚未保存的内容；旧保存任务不能在恢复之后突然写回来。

所有数据仍然只留在浏览器本地。本阶段不增加服务器、账号、社区、上传、云同步、遥测或远程数据库。

三个词先说清楚：

- **编辑锁**：浏览器在本机协调多个标签页，同一作品一次只给一个页面编辑；
- **候选数据库**：包含本次最新修改、但还没有确认写入成功的完整作品库；
- **精确读回**：写入后马上重新读取，内容一字不差并再次校验通过，才算真正保存。

## 背景

Tuuru 把完整创作库保存在一个 `localStorage` 键 `tuuru_works` 中。当前 `js/data.js` 的普通修改流程是：读取并校验整库、修改一份对象、再次校验，然后同步重写整库。

这带来几个实际问题：

- 文章正文的每次 `input` 都会触发一次整库读写；
- 普通写入没有统一的脏状态、保存状态或错误恢复界面；
- 路由跳转、`refreshEditor()`、页面隐藏和关闭前没有统一刷新入口；
- 手机编辑器常常持有一份较早的 `phoneData`，可能覆盖后来发生的修改；
- 同一作品可以在多个标签页中同时编辑；
- 选项批量保存、章节删除以及手机模块卡片创建/删除由多次独立写入组成，中途失败会留下不完整数据；
- 完整备份和恢复已经具备严格校验，但普通编辑保存还没有同等级别的保护。

## 目标

- 大幅减少正文输入产生的整库写入，同时保留现有编辑能力。
- 保存成功必须代表 `setItem()` 成功、精确读回一致并再次通过数据库校验。
- 能够证明没有写入的失败必须保持旧存储不变；无法确认结果时保留候选并停止写入，不静默猜测。
- 同一作品在同一时刻只有一个有效编辑者。
- Tuuru 自己对 `tuuru_works` 的全部读—改—写在浏览器同源数据库写锁内串行执行。
- 不同作品的标签页从最新整库开始合并自己的修改，不使用长期持有的过期整库快照。
- 用户可见地处理存储配额、权限、校验、冲突和编辑权丢失。
- 把已知的多步骤操作改成单次原子提交。
- 完整备份恢复与普通编辑保存互斥，恢复后不存在旧任务回写。
- 每一项改动都可以通过一个小型 Git 提交单独回退。

## 不做的事情

- 不迁移到 IndexedDB，不引入 CRDT、服务工作线程或框架重写。
- 不增加任何联网存储、上传、社区、账号、云同步或遥测。
- 不改变单篇作品 JSON/PNG 导出的分享用途。
- 不自动删除、压缩或修改用户旧作品来腾出空间。
- 不在这一阶段重做编辑器和手机模块的整体视觉设计。
- 不把所有页面改造成一套常驻内存的全局数据库。
- 不自动保存尚未点击“保存”的不完整表单。此类表单会获得离开保护，但只有通过表单自身校验并确认后才进入作品数据。
- 不使用 `localStorage` 心跳记录冒充真正的互斥锁；心跳只负责说明占用者和判断是否允许人工接管。

## 方案选择

### 方案一：有针对性的保存协调器——采用

只合并高频正文输入；结构性操作立即提交。所有编辑器写入经过同一个作品级协调器，并使用最新本地数据库作为每次提交的基础。

优点是直接解决当前最严重的问题，且不需要一次重写全部数据层。现有纯本地设计、作品格式和阅读器行为都能保留。

### 方案二：完整内存数据库——暂缓

让所有页面共同编辑一份常驻内存数据库，再批量持久化。长期看很整洁，但会同时改变 `data.js`、手机草稿、导入恢复、返回值和大量同步调用的语义，风险接近一次大重构。

### 方案三：迁移 IndexedDB——拒绝用于本阶段

IndexedDB 提供更大的容量和事务能力，但不能自动解决脏状态、离开保护、同作品多标签冲突和用户错误提示。迁移和回滚成本明显高于当前收益。

## 核心原则

1. **暂存在页面里不等于已经保存。** 只有精确读回并校验成功才能显示“已保存”。
2. **每次提交从最新整库开始。** 不把打开编辑器时读取的整库快照直接写回。
3. **一次用户操作只产生一个候选数据库。** 原子操作全部成功，或完全不写。
4. **失败内容继续留在协调器中。** 明确没有写入时可以重试；结果未知时只能重新检查或生成紧急备份。
5. **无法确认写入结果时不猜。** 停止继续写入，要求重新检查或重新加载。
6. **编辑权是合作保护，不是加密或安全边界。** 它防止 Tuuru 自己的多个标签页互相覆盖，但不能阻止开发者工具或其他脚本直接修改 `localStorage`。

## 组件设计

### 1. 浏览器本地锁层级

真正的互斥由浏览器的 Web Locks API 提供；`localStorage` 心跳记录只负责显示占用者和判断人工接管条件。锁全部属于同源浏览器环境，不联网。

实现以 [W3C Web Locks 规范](https://w3c.github.io/web-locks/) 的独占/共享锁、`ifAvailable`、`steal` 和页面终止释放语义为准，不自行模拟一套不可靠的互斥协议。

固定使用三个层级，并始终按以下顺序申请，避免死锁：

1. `tuuru:library-session`：编辑器整个会话持有共享锁；完整恢复持有独占锁。
2. `tuuru:work:<encoded-work-id>`：真实作品编辑器整个会话持有独占锁。
3. `tuuru:database-write`：每次修改 `tuuru_works` 时短暂持有独占锁。

这样，不同作品可以同时保持编辑会话，但实际整库写入会逐次排队。同一作品的第二个编辑器使用 `ifAvailable` 检查，拿不到作品锁就不渲染可编辑页面，并立即释放它为检查而取得的共享 library-session 锁。页面正常关闭或崩溃时，浏览器负责释放所持锁。

创建、复制、删除作品以及完整恢复等首页写操作也必须经过相同锁层级，不能继续绕过数据库写锁直接改 `tuuru_works`。

删除作品、修改已有作品信息和复制作品都必须先用 `ifAvailable` 取得目标作品锁；目标正在编辑时，首页显示占用提示并拒绝操作。复制不会偷偷复制编辑器内存中的未保存状态，只在取得目标锁后复制最新已落盘版本。创建作品会先分配新 ID，再取得该新 ID 的作品锁并执行数据库写入。

Web Locks 只在安全上下文可用。如果当前浏览器环境没有可靠的 Web Locks，Tuuru 保留读取、阅读、导出和恢复材料下载能力，但阻止编辑及其他写操作，并明确说明浏览器环境不支持安全编辑。不会悄悄降级成可能覆盖数据的写法。

锁实现包装成可注入适配器，单元测试使用可控假锁，不依赖测试运行器真实提供 Web Locks。

### 2. 普通数据库原子提交

在 `js/storage.js` 的校验能力之上增加异步普通修改入口。概念接口如下：

```js
await commitLocalDatabaseMutation({
  workId,
  operationId,
  apply(database),
}, {
  storage,
  lockManager,
  assertWorkOwner,
})
```

`apply()` 只修改传入的候选数据库，不读取或写入存储。创建 ID 的操作必须在进入提交前分配 ID，使一次明确未写入的失败可以安全重试而不产生重复记录。

一次提交在 `tuuru:database-write` 独占锁内按以下顺序执行：

1. 确认当前编辑会话仍持有正确的作品所有者令牌和启动时记录的恢复代次。
2. 精确读取当前 `tuuru_works` 原文并通过现有深层校验。
3. 基于这份最新数据库创建候选数据。
4. 对候选数据应用一次完整修改，并更新作品的 `updatedAt`。
5. 校验并序列化完整候选数据库。
6. 在 `setItem()` 前再次确认作品所有者令牌、恢复代次和当前原文；若被外部代码改变，则不写入并报告冲突。
7. 调用一次 `setItem()`，不会先调用 `removeItem()`。
8. 精确读回；只有字节一致且再次通过数据库校验才返回成功。

第 2 步读取的完整原文同时充当版本令牌，不需要给作品导出格式增加内部版本字段。数据库写锁负责串行化 Tuuru 自己的标签页；精确原文检查继续防守开发者工具、旧版本页面或其他未遵守锁合同的代码。

普通修改错误沿用 `LocalDatabaseError`，并提供稳定代码和提交状态：

- `mutation-conflict`：写入前发现版本改变，没有写入；
- `mutation-write-failed`：`setItem()` 失败，旧原文保持不变；
- `mutation-readback-failed`：写入后无法读取，最终状态未知；
- `mutation-verification-failed`：读回不一致或校验失败，最终状态未知；
- `mutation-lease-lost`：当前标签页已无编辑权，没有写入；
- `mutation-lock-unavailable`：无法取得可靠本地锁，没有写入；
- `mutation-invalid`：候选数据无效，没有写入。

只有明确为“没有写入”的错误可以重试。读回结果未知时绝不把同一批修改再次送入写入队列。

### 3. 作品保存协调器

每个真实作品编辑会话只创建一个协调器。文章编辑器、独立手机编辑器以及文章内部的手机模块共同使用这个作品的协调器。

协调器状态为：

```text
clean -> dirty -> saving -> clean
                    |       |
                    |       v
                    |     dirty（保存期间又有新输入）
                    v
       error-retryable / error-invalid / error-unknown

任意状态 -> conflict / lease-lost -> disposed
```

协调器提供三类能力：

- `stage(key, mutation)`：同步放入可合并修改；相同 `key` 的新正文替换旧正文；
- `commitNow(mutation)`：异步把已有待保存修改和结构性操作合成一次提交；即使调用时已有保存进行中，也必须等到这项结构修改本身确认提交后才完成；
- `flush(reason)`：异步保存当前已经形成的一批修改；
- `drain(reason)`：记录当前修改代次，等待进行中的批次，然后循环保存，直到没有更新代次、未知批次或新待处理修改。并发 `drain()` 共享同一个排空任务。

正文使用 600ms 尾随延迟。用户持续不停输入时，最迟每 3 秒主动刷新一次，避免长时间一直处于未保存状态。

错误代码到状态的映射固定如下：

- `mutation-write-failed` → `error-retryable`：修改继续留在待处理队列，之后的新输入可以合并；
- `mutation-invalid` → `error-invalid`：不写入，只有用户修正候选后才允许再次保存；
- `mutation-conflict` → `conflict`；
- `mutation-lease-lost` → `lease-lost`；
- `mutation-lock-unavailable` → 只读失败关闭；
- `mutation-readback-failed` 或 `mutation-verification-failed` → `error-unknown`。

读回结果未知时，正在提交的批次移入独立的 `uncertainBatch` 并进入 `error-unknown`。协调器暂停计时器、自动保存和后续编辑。“重新检查”精确读取当前原文：

- 等于候选原文：确认本批已经保存；
- 等于写入前原文：确认没有写入，将批次放回队列并允许重试；
- 两者都不相等：进入 `conflict`，只允许下载紧急备份并重新加载。

每次 `stage()` 都增加修改代次。保存期间产生的新修改属于下一批，上一批成功不能把整个状态错误标成 `clean`；路由清理和模态关闭必须使用 `drain()`，不能只等待较早的一次 `flush()`。

待保存修改在进入队列时转成不可变快照并获得单调递增序号。批次按以下规则组合：

1. 先按序号应用普通暂存字段替换；
2. 再应用本次结构性操作；
3. 结构操作如果也修改某个已暂存键，必须声明 `consumes: [key]`，读取已经应用暂存修改后的值，并由结构操作产出该键的最终值；
4. 被消费的旧暂存项不能在下一批再次回放。

例如创建手机卡片时，先取得包含最新未保存正文的 `node.content`，再在这份内容中插入卡片并消费正文暂存键；不能让稍早的整段正文在卡片保存后把卡片覆盖掉。

`phone-draft:*` 继续同步更新现有内存草稿，不申请作品锁，也不直接写入本地数据库。草稿在明确保存或外层关闭提交时，才把一个完整修改交给真实作品协调器。

因为真实作品提交现在需要等待数据库写锁，模态关闭控制器变为 Promise 感知的单次流程：保存期间模态保持打开并阻止重复关闭；成功后才移除；失败仍然否决关闭、恢复焦点并允许重试。虚拟草稿的更新和 `snapshot()` 仍保持同步。

### 4. 同作品独占编辑与不同作品协调

作品独占权由长时间持有的 `tuuru:work:<id>` Web Lock 决定。本地占用记录不进入作品备份，也不会发送到网络，只保存用于界面说明的信息：

```js
{
  version: 1,
  workId,
  ownerId,
  leaseId,
  heartbeatAt,
  expiresAt,
}
```

- `ownerId` 和 `leaseId` 使用当前标签页生成的随机值；
- 默认每 15 秒更新一次占用记录，60 秒没有有效心跳后才允许显示人工接管；
- 页面隐藏前更新一次记录，重新显示时验证作品锁和所有者令牌；
- 正常离开时只清理自己的记录并释放自己的锁；
- 作品锁不可用时，第二个标签页只显示占用说明、返回作品列表和重新检查；
- Web Lock 已经自动释放但旧记录仍在时，可以直接取得锁并替换旧记录；
- Web Lock 仍被持有且记录确实过期时，只有用户明确确认后才使用 `steal` 接管，绝不自动抢占；
- 接管前不能改所有者令牌。确认取得被 `steal` 的作品锁后，再取得 `tuuru:database-write`，等待旧写入结束，写入并精确读回新所有者令牌，然后从最新数据库初始化协调器；全部成功后才渲染编辑器；
- 初次登记、替换陈旧记录、普通接管、`steal` 接管、续租和释放等每一次所有者令牌变化，都必须在已经取得作品锁后短暂持有 `tuuru:database-write`；这样接管前已经在途的提交会先结束，之后的旧所有者提交都会验证失败；
- 接管任一步失败都释放本页已取得的资源并保持只读，绝不恢复旧令牌或假装取得编辑权；
- 当前页面收到锁终止、所有者记录变化或数据库外部变化时，立即停止写入并保留恢复候选。

不同作品各自持有不同作品锁和同一个共享 library-session 锁，因此可以分别编辑。它们的保存通过 `tuuru:database-write` 串行执行，每次都从最新整库应用字段级修改，不会拿打开页面时的旧整库互相覆盖。

监听 `storage` 事件仍然用于发现恢复、旧版本页面或开发者工具绕过锁的外部变化。若当前作品回退、消失或出现未知改变，协调器进入 `conflict`，而不是猜测哪一份应该覆盖另一份。

### 5. 路由与页面生命周期

路由增加可选的“离开前检查”和“页面销毁”生命周期：

```js
registerNavigationGuard(guard)
registerRouteCleanup(cleanup)
```

路由只有一条串行转换管线。`currentRoute`、参数和页面 DOM 只在离开检查成功后更新：

- `navigate()` 把目标交给转换管线，不直接清空页面；
- 浏览器 Back/Forward 或手工改 hash 时，URL 可能已经先变化，管线仍保留当前 DOM；拒绝导航时用 `history.replaceState` 恢复最后一次已接受 URL，并抑制这次恢复产生的重复处理；
- 离开检查进行中再次收到导航，只保留最新目标，不并行运行第二次清理；
- 成功时只执行一次旧页面 cleanup，再渲染最终目标；失败时当前路由状态、参数、DOM 和焦点保持不变。

离开流程为：

1. 如果输入法仍在组合文字，只记录最新待导航目标，继续允许输入法正常结束，不能提前禁用编辑器或销毁 DOM；
2. 收到 `compositionend` 后暂存最终文字，再暂停接受新的编辑输入；
3. 检查仍未提交的表单草稿；选择保存时完成表单校验和提交，选择继续编辑时取消导航，只有明确放弃才清除草稿；
4. 要求当前协调器 `drain("navigation")`，保存并循环处理保存期间新产生的批次；
5. 再次确认协调器没有待处理代次、未知批次或脏表单；
6. 成功后释放事件、计时器和本地锁；
7. 最后清空旧 DOM 并渲染新路由。

如果保存失败或表单仍未处理，保留当前 DOM 和当前路由，不先清空页面。

生命周期规则：

- 只有正文编辑区和明确配置为“失焦保存”的字段发生 `blur` 时才刷新，不把任意控件或窗口失焦都变成整库写入；
- `visibilitychange` 变为隐藏和 `pagehide` 时发起尽力刷新；Web Locks 是异步 API，浏览器仍可能在完成前终止页面；
- 只要协调器仍有可恢复的未保存修改，或任一注册表单草稿为脏，就安装浏览器原生 `beforeunload` 警告；这也包括带未保存候选的 `lease-lost`；
- `refreshEditor()` 等同一路由的整页重绘，也必须先刷新或把待保存修改带入新 DOM；
- 手动删除分隔线等 DOM 修改要立即进入协调器，不能等待下一次无关输入才保存。

中文、日文等输入法组合期间，`input` 可以更新页面内存候选，但不能启动保存计时器或保存中间拼音/假名；`compositionend` 后才以最终文字重新暂存并启动 600ms 计时。组合状态本身属于未解决的本地修改，关闭或刷新时也触发 `beforeunload`。

浏览器进程突然崩溃或设备断电时，网页无法执行离开逻辑。理论上最后约 600ms 的正文可能丢失，这是合并保存的明确代价。

### 6. 表单草稿

选项、占位符和手机设置等明确带“保存/取消”的面板仍保持显式保存，不把用户尚未确认的半成品自动写入作品。

这些面板需要注册一个本地草稿状态：

- 未修改时可以直接关闭；
- 修改后关闭、刷新或路由离开时要求保存、放弃或继续编辑；只有表单草稿也必须触发 `beforeunload`；
- 选择保存后必须通过表单校验、等待原子提交并再次 `drain()`，之后才允许离开；
- 选择继续编辑会取消本次导航；只有明确选择放弃才清除草稿；
- 未确认的 DOM 表单内容不声称已经包含在完整紧急备份中。

## 已知多步骤操作的原子化

### 选项批量保存

不再循环执行“删除旧选项、添加新选项、逐条修改文字”。保存按钮先构造完整新选项数组并分配稳定 ID，然后在一次候选数据库修改中替换目标节点的 `choices`。

### 章节删除

一次修改同时：

- 删除目标章节；
- 将该章节节点移动到第一个剩余章节；
- 如果没有剩余章节，则把这些节点的 `chapterId` 清空，避免指向不存在的章节；
- 更新作品时间。

任何校验或写入失败都不会只移动部分节点。

### 手机模块卡片创建和删除

创建时，手机模块记录和文章节点中的内联卡片 HTML 在同一个候选数据库里提交。提交成功后才完成 DOM 更新；失败时模态编辑器保持可重试状态。

删除时，模块记录和文章节点中的对应卡片也在一次提交中移除。当前代码存在的两次写入风险一并消除。

编辑已有模块的数据本身仍是一次普通原子修改。

独立手机编辑器中的面板不能再把打开面板时取得的整份旧 `phoneData` 当作最终真相。每个面板只提交自己负责的集合或字段；如果它依赖的基线已经变化，则显示冲突并保留草稿，不覆盖协调器中更晚的修改。

真实作品使用明确的字段级接口：

```js
await mutateStoredPhoneWork(workId, {
  scope,
  expectedScopeToken,
  apply(phoneData),
})
```

`apply()` 在最新 `phoneData` 上只修改声明的 `scope`。面板打开时记录该 scope 的基线令牌；提交时若同一 scope 已改变则冲突，不执行覆盖。真实作品边界拒绝 `{ phoneData: wholeSnapshot }` 形式的整对象替换；该同步浅补丁行为只保留给 `phone-draft:*`。

## 保存状态与错误界面

保存状态复用现有颜色、字体和按钮组件，不进行整体视觉改版。

- 桌面端放在编辑器顶部，靠近字数或操作区；
- 窄屏显示简短图标和文字，不挤压作品标题；
- 独立手机编辑器的状态属于编辑器外壳，不放进作品中的虚拟手机画面；
- 普通成功状态使用 `role="status"`；持久错误使用 `role="alert"`；
- `role="status"` 只播报首次进入未保存、最终保存完成和错误恢复，不在每次自动保存时反复朗读；
- 状态不只依靠颜色表达，触摸操作至少 44px；
- 动画只用于短暂状态变化，并服从 `prefers-reduced-motion`。

普通状态文案：

- `未保存`
- `正在保存…`
- `已保存`

保存失败使用不会自动消失的内联提示，不使用几秒后消失的 toast。明确未写入的错误提供：

- `重新保存`
- `下载紧急备份`
- `放弃修改并离开`

“放弃修改”需要二次确认。不同错误的动作固定如下：

| 状态 | 大白话提示 | 可以做什么 | 可直接重试 |
| --- | --- | --- | --- |
| 空间不足或写入被拒绝 | 最新修改还在本页，但没有写入浏览器 | 重新保存、下载紧急备份、明确放弃 | 是 |
| 候选校验失败 | 修改没有写入；当前作品数据不符合保存要求 | 下载最近一次有效完整备份；草稿若能安全序列化，可另存为“不可直接恢复”的未验证草稿 | 修正后才可 |
| 写入结果未知 | 浏览器无法确认刚才是否写入成功 | 重新检查、下载紧急备份 | 否 |
| 数据冲突 | 本地创作库已被其他页面改变，本页已经停止保存 | 下载紧急备份、重新加载 | 否 |
| 编辑权丢失 | 此作品已由另一个标签页接管，本页已经停止保存 | 下载紧急备份、返回作品列表 | 否 |
| 浏览器不支持可靠锁 | 当前环境不能保证本地数据不会互相覆盖 | 保持只读、导出已有内容、返回作品列表 | 否 |

编辑权丢失后页面变为只读，提供紧急备份、返回作品列表，以及在租约过期后重新接管。有效租约存在时不能强行抢占。

## 紧急完整备份

紧急备份使用现有完整创作库备份格式，而不是单篇作品导出格式。它包含：

- 全部作品；
- 全局集合；
- 密码、编辑器设置和作者私有字段；
- 当前协调器中尚未成功落盘但已经进入有效作品数据的修改；
- 现有未知字段和未来兼容字段。

它不包含其他标签页仍在内存、尚未落盘的修改，也不包含用户尚未确认的表单草稿。检测到其他活动编辑会话时，界面必须持续提醒每个标签页分别处理自己的未保存内容。

生成备份时优先重新读取当前最新的有效本地数据库，再应用当前全部待保存修改；只有当前存储无法读取时，才退回协调器保存的最近一次有效基线。最终候选必须再次通过完整数据库校验。文件名类似：

```text
tuuru-emergency-backup-2026-07-12T14-30-00-000Z.json
```

界面必须说明该文件包含私密编辑数据，只用于恢复，不应作为单篇作品分享。浏览器只能确认已经发起下载，不能保证文件已经永久写入磁盘。

下载紧急备份不会把状态改成“已保存”。

如果待保存候选本身无法通过数据库校验，就不能把它包装成“可恢复完整备份”。此时完整备份只包含最近一次有效数据库；若未验证草稿仍可安全序列化，可以另行下载并明确标注“仅供人工找回内容，不可直接恢复”。如果连安全序列化也失败，界面必须如实说明无法生成该草稿文件。

如果当前状态已经是同作品冲突或编辑权丢失，备份不能用本页版本覆盖存储中的另一版本。它会保留最新存储作品，并把本页候选复制成一个预先分配新作品 ID、带“冲突恢复副本”标题和恢复来源元数据的第二份作品。这样恢复该完整备份后两份内容都在，用户可以自行比较，不由程序替用户选胜者。

`error-unknown` 生成备份时也不能重新执行 `apply()`。协调器保留已经验证过的 `candidateRaw` 和写入前的 `expectedCurrentRaw`，并先执行与“重新检查”相同的精确判断：

- 当前原文等于 `candidateRaw`：本批已经保存，直接从当前数据库生成备份；
- 当前原文等于 `expectedCurrentRaw`：本批没有写入，直接从保存下来的 `candidateRaw` 生成备份，不重放操作；
- 当前原文是第三种有效数据：保留该当前数据库，并把 `candidateRaw` 中的本页作品作为冲突恢复副本附加；
- 当前原文无法读取：以写入前的有效完整数据库为基线，附加 `candidateRaw` 中的本页恢复副本，并明确标注无法确认浏览器当前存储状态。

任何未知状态备份路径都不得再次调用不确定批次的修改函数，避免重复追加节点、选项或模块。

## 与完整备份恢复的关系

- 每个编辑会话持有共享 `tuuru:library-session` 锁；完整恢复必须取得同名独占锁；
- 任一编辑会话仍在运行时，恢复不能取得独占锁并立即中止，不排队等待用户忘记关闭的页面；
- 恢复持有独占 library-session 锁期间，新编辑器无法取得共享锁；
- 恢复的实际 `setItem()` 还必须持有 `tuuru:database-write`，与全部普通写入使用同一个可靠互斥边界；
- 编辑会话启动时记录持久化的恢复代次。恢复在独占 library-session 和 database-write 锁内先生成、写入并精确读回新的代次，再替换数据库；每次普通提交都必须确认代次未变；
- 从浏览器后退缓存或暂停状态恢复的旧页面必须重新取得锁并检查恢复代次，代次不一致就销毁旧协调器，不能继续写入；
- 恢复成功后广播数据库替换事件并重新加载；
- 所有旧协调器、延迟计时器和待提交批次都失效，不能在恢复之后回写；
- 现有恢复“准备—精确冲突检查—单次替换—读回验证”合同保持不变。

## 测试策略

### 原子提交

- 从最新数据库应用修改，不写长期持有的过期整库快照；
- 全部 Tuuru 写入在同一个可注入数据库 Web Lock 中串行执行；
- 外部代码导致写入前版本改变时，本次不写入并报告冲突；
- 明确未写入后的重试使用预先分配的 ID，不产生重复记录；
- 无效候选、租约丢失和冲突耗尽时不调用 `setItem()`；
- 配额或权限错误保持旧原文完全不变；
- 成功只在一次 `setItem()`、精确读回和重新校验后返回；
- 读回异常或不一致不自动重试或回滚。
- 低层同步 `writeLocalDatabase()` 不再向普通页面暴露绕过锁的写入口；静态合同测试阻止编辑器、手机页和首页直接导入它。

### 保存协调器

- 使用假计时器证明 600ms 合并和 3 秒最长等待；
- 相同正文键只保留最后内容；
- 结构性操作把待保存正文合并进同一次提交；
- 并发 `flush()` 不会重复写入；`drain()` 会继续处理等待期间出现的下一批，直到代次稳定；
- 导航开始后出现的最后输入不能在页面清理时被丢弃；
- 明确未写入的失败保留队列并可重试；
- 未知批次被隔离并冻结后续编辑；重新检查只会确认成功、确认未写入或进入冲突，绝不盲目重放；
- 错误代码分别进入 retryable、invalid、unknown、conflict、lease-lost 和 lock-unavailable，不能把所有未写入错误都当作可直接重试；
- `lease-lost` 和 `disposed` 后拒绝新写入；
- 失败候选可以生成包含最新有效修改的完整备份。

### 编辑锁与多标签页

- 同一作品第二个标签页不能获得独占作品锁；
- 不同作品可以分别获得作品锁，同时共享 library-session 锁；
- 正常关闭和模拟崩溃会释放 Web Lock；
- 心跳记录、过期判断和显式 `steal` 接管使用可注入时钟及假锁测试；
- 接管后旧所有者令牌不能再通过数据库写入检查；
- 失去作品锁或所有者令牌立即冻结协调器；
- Web Locks 不可用时所有写操作失败关闭，已有读取和导出保持可用；
- 真实浏览器双页面验证在安全的 localhost 上下文运行，覆盖 `ifAvailable`、不同作品数据库写锁串行、显式 `steal`、关闭页面自动释放、暂停/BFCache 恢复和无 Web Locks 时失败关闭；
- 单元测试继续使用假锁验证可重复的队列和异常分支；真实浏览器验证使用仓库内轻量测试页/脚本，若必须新增浏览器测试依赖，实施前单独评估并记录，而不把依赖偷偷混入功能提交；
- 数据库 `storage` 事件不会把其他作品的正常变化误报成当前作品冲突；
- 当前作品被外部回退、删除或改变时停止写入并保留恢复候选。
- 冲突紧急备份保留存储版本，并以新作品 ID 附加本页恢复副本，不覆盖任一分支。
- 活动编辑器持有目标作品锁时，首页删除、修改和复制都失败关闭；释放后才能继续。
- 接管发生时若旧页面已有数据库提交在途，该提交先结束，新令牌再登记；登记后的旧提交全部失败。

### 文章编辑器

- 高频正文输入不再每个字符写整库；
- 输入法组合期间不保存中间文本，`compositionend` 后保存最终文字；
- 切换节点、刷新编辑器和路由离开前刷新；
- 保存失败时路由和 DOM 保持不变；
- 手动删除分隔线立即进入待保存状态；
- 选项替换、章节删除和手机模块卡片创建/删除各只产生一次数据库提交；
- 任何原子操作失败都不留下部分结果。
- 未保存正文与手机卡片创建/删除同时发生时，批次先应用正文，再由结构操作消费最新 `node.content`，两边内容都保留。

### 手机编辑器

- `phone-draft:*` 继续同步更新独立内存草稿；
- 真实作品更新进入同一个作品协调器；
- 显式保存成功前不显示成功 toast；
- 模态关闭失败仍能否决关闭并恢复焦点；
- 拖拽位置只有在有效落点改变时提交一次，取消拖拽绝不提交；
- 面板按自己负责的字段或集合提交；旧 `phoneData` 快照不能覆盖协调器中更晚的同作品修改。
- 真实作品拒绝整份 `phoneData` 快照替换；虚拟草稿继续支持现有同步浅补丁。

### 路由、恢复和界面

- 应用内导航等待刷新成功；失败时不清空当前页面；
- 仅有脏表单草稿时也安装 `beforeunload`；选择保存会提交并排空，选择继续编辑取消导航，选择放弃才清除；
- 仍有可恢复未保存修改时安装 `beforeunload` 警告，包括编辑权丢失后的候选；
- 完整恢复被活动编辑会话阻止；
- 恢复持有独占 library-session 锁时不能启动新编辑会话；恢复写入与普通写入共享 database-write 锁；
- 恢复代次改变会使后退缓存或暂停后恢复的旧协调器失效；
- 恢复后旧计时器不能写回；
- 错误提示不自动消失，按钮可通过键盘操作并具有可读名称；
- 在 320px、375px 和桌面宽度检查状态、错误按钮、软键盘和安全区布局。

每个实现提交先运行聚焦测试，再运行完整 `npm run verify`。验证完成后 Git 可见工作区必须保持干净。

19. 旧写入口封闭：静态检查确认文章、手机、首页和恢复都不再绕过数据库写锁。
20. 统一启用：用一个小提交同时启用文章和手机的新写入路径，并完成两者回归。
21. 回归：真实双页面锁、输入法、窄屏、可访问性和完整构建检查。

第 3 至第 10 步先完成全部基础能力和恢复隔离，不把半套保存流程接到生产编辑器。文章第 12 至第 15 步、手机第 16 至第 18 步都保持同一个生产旗标关闭；第 19 步证明所有 `tuuru_works` 写入口均已迁移后，第 20 步才同时启用文章和手机。绝不允许新协调器与旧同步写入器在生产路径中并存。

如果任何一步有害，只回退对应的原子提交，不重写 Git 历史，也不手工撤销整个阶段。

## 成功标准

- 正文输入不再每个字符重写完整创作库。
- 支持 Web Locks 的安全浏览器环境中，同一作品无法在两个标签页中同时编辑；不支持时写操作失败关闭。
- 不同作品可以分别编辑，但全部整库写入被可靠串行化；外部绕过锁造成的变化会明确停止并保留候选。
- 选项、章节和手机模块卡片操作不会留下半完成状态。
- 用户始终能知道内容是未保存、保存中、已保存、失败还是失去编辑权。
- 明确未写入的错误可重试；未知提交状态不会被伪装成成功。
- 紧急备份可包含已经进入协调器但尚未落盘的最新有效作品数据，并兼容完整恢复流程。
- 完整恢复后不存在旧编辑任务回写。
- 文章阅读器、作品导出、虚拟手机草稿及现有移动端交互保持可用。
- 不产生任何服务器、上传、远程数据库、遥测或新增网络行为。
- 每个代码提交的聚焦测试和 `npm run verify` 均通过，且提交前后工作区干净。
