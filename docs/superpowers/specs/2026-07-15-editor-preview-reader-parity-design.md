# 编辑器文章预览与真实阅读器统一设计

## 背景与目标

Tuuru 当前存在两套文章阅读实现：作者应用中的 `/read/:id` 使用 `js/pages/reader.js`，独立阅读器使用 `reader/reader.js`。文章节点会把作者态手机模块卡片 HTML 存进 `node.content`，而旧预览器直接输出这段 HTML，因此读者预览会显示“编辑/删除”卡片。编辑器还把点击事件委托永久挂在 `document` 上，先编辑再预览时，残留的作者状态会让预览中的卡片继续打开作者编辑界面。

本阶段只修复这一项 P0：作者点击“阅读”后，直接进入与导出作品相同的真实阅读器运行时。正确体验必须是“剧情提示 → 确认接入指定角色的手机 → 查看角色内容”，且预览中不能出现任何作者编辑控件。

## 已确认根因

- `js/pages/reader.js` 只做占位符与换行替换，然后把 `node.content` 原样注入页面。
- 作者态 `.pm-inline-card`、汉堡按钮和 `data-a="pm-hamburger"` 因而完整进入预览。
- `js/pages/editor.js` 的 `document` 级事件委托没有路由卸载边界，模块级作品与节点状态也会跨路由保留。
- 独立阅读器已经具备手机模块触发器、未读状态、读者手机 overlay、角色接入确认、失效角色安全关闭以及对应测试。

根因不是手机模块数据缺失，而是作者应用维护了一个没有消费真实读者模型的重复阅读器。

## 方案比较

### 方案 A：作者预览直接启动真实阅读器（采用）

把作者应用中的 `/read/:id` 改为薄跳转桥接，进入随编辑器构建发布的真实阅读器页面。阅读器通过明确的预览查询参数，从同源本地创作库只读取得指定作品，然后走现有真实阅读器入口。

优点：读者行为只有一套实现；作者事件不会进入阅读器页面；现有手机剧情、选项和排版能力自然保持一致；后续真实阅读器增强会自动进入作者预览。

代价：预览是一次同页全屏页面切换；编辑器构建需要同时包含阅读器 HTML 入口。

### 方案 B：抽出可嵌入 SPA 的共享阅读器运行时（不采用）

把当前三千多行、带全局事件和启动副作用的阅读器拆成可挂载组件，再嵌入作者应用。

优点：可以保留作者应用外壳。代价：需要同步拆分运行时、样式与全局生命周期，明显扩大 P0 风险面，并可能引入 CSS 和事件冲突。

### 方案 C：把手机模块逻辑复制回旧预览器（拒绝）

短期改动较小，但继续保留两套阅读逻辑，无法满足“直接沿用真实阅读器”，并会在论坛、通话、消息和后续功能上再次漂移。

## 架构决策

### 作者端薄桥接

- 保留现有 `/read/:id` 地址作为兼容入口。
- 该路由不再渲染文章、密码、占位符、选项或手机模块。
- 它只构造同源 `reader/index.html?preview=<work-id>` 地址，并使用同页替换进入真实阅读器。
- URL 使用标准 `URL` 与 `URLSearchParams` 构造，作品 ID 不通过字符串拼接进入 HTML。
- 旧的 `js/pages/reader.js` 改为只负责预览 URL 与跳转，不再保留第二套阅读器实现。

### 编辑器构建中的真实阅读器入口

- `vite.config.ts` 把 `reader/index.html` 加入编辑器的多页面构建输入。
- 正式编辑器产物因此包含真实阅读器页面及其实际依赖；不依赖开发服务器恰好暴露源码目录。
- 现有独立阅读器构建继续使用 `vite.reader.config.ts`，两种产物引用同一个 `reader/reader.js` 源码。
- 不改变现有 `build`、`build:editor`、`build:reader`、`preview` 或 `preview:reader` 命令语义。

### 阅读器预览启动

- 普通打开阅读器时仍显示个人主页与导入入口。
- 存在有效 `preview` 参数时，阅读器从 `tuuru_works` 中只读查找对应作品。
- 本地数据库读取沿用现有完整性校验；作品不存在、数据库损坏或不可访问时显示安全错误状态，不回退到其他作品。
- 找到作品后，直接进入现有密码/占位符起始页，再调用现有文章或手机阅读运行时。
- 预览作品只存在于本次页面内存中，不写回 `tuuru_works`。
- 预览不写入 `moirain_work_<id>`，也不加入读者最近记录；读者自己的排版、美化与占位符偏好仍可照常使用。

### 返回行为

- 浏览器返回可回到进入预览前的作者页面。
- 真实阅读器中的“返回首页”在预览模式下回到同源作者首页，而不是进入读者导入首页。
- 直接打开一个失效预览 URL 时仍提供明确的返回作者首页入口。

## 数据流

1. 作者在作品列表点击“阅读”。
2. Hash 路由解析并验证作品 ID 的 URL 表达。
3. 作者应用使用 `location.replace` 进入同源阅读器预览 URL。
4. 阅读器读取并校验 `tuuru_works`，只选择 ID 精确匹配的作品。
5. 阅读器在内存中启动现有 `loadWork` 流程，但关闭作品缓存和最近记录写入。
6. 文章正文中的作者卡片由现有真实阅读器转换成 `.rd-pm-trigger`。
7. 点击触发器后沿用现有手机 overlay、角色接入确认和 App 内容运行时。

作者应用不再解释 `node.content` 中的手机卡片，也不再拥有任何读者手机交互分支。

## 安全与失败处理

- 预览作品 ID 只用于精确查找，不回显到错误 HTML。
- 本地数据库解析、结构和作品格式继续走现有校验边界。
- 找不到指定作品时 fail closed：不选择第一篇作品，也不展示其他本地作品。
- 预览不新增跨窗口消息、Blob URL、内联 JSON、跨域存储或远程上传。
- 真实阅读器现有的富文本清洗、图标清洗、图片策略、失效选项和失效角色保护保持生效。

## 自动化测试

先建立会失败的回归测试，再实现桥接：

1. 预览 URL 测试：子目录部署下仍指向同源 `reader/index.html`，并正确编码作品 ID。
2. 路由契约测试：`/read/:id` 只启动真实阅读器，不再调用重复的 `renderReader`。
3. 编辑器构建契约测试：编辑器 Vite 输入包含 `reader/index.html`。
4. 阅读器预览启动测试：从 `tuuru_works` 精确加载指定作品，跳过读者作品缓存和最近记录。
5. P0 交互回归：带角色绑定备忘模块的文章进入预览后，只显示读者触发器；点击后先显示剧情接入提示，确认后才显示指定角色内容；页面中不存在 `.pm-inline-card` 或作者编辑控件。
6. 失败状态：作品缺失或创作库损坏时不泄漏其他作品。
7. 现有独立阅读器手机模块、角色接入、选项、Schema 与传输测试继续通过。

完成后运行相关 Node 测试、全量 `npm test`、TypeScript 与两个 Vite 临时构建验证。

## 手动验收

- 从作者作品列表打开一篇含手机模块的文章。
- 预览占满当前页面并使用真实阅读器视觉，不显示作者应用 Header 或编辑卡。
- 正文位置显示剧情手机提示按钮。
- 点击提示后打开手机；进入已绑定的备忘、相册、浏览器或购物 App 时，先看到指定角色与剧情提示。
- 取消后不泄漏内容；确认后只显示绑定角色内容。
- 返回可回到作者端，作品内容与更新时间未因预览发生变化。
- 刷新预览不会把作品加入读者最近记录。

## 范围边界

本阶段不处理消息保存后自动预览、论坛发帖/回复、通话逐句播放、回复气泡细分、拖拽排序、默认节点分章、整体字号与对比度、阅读器空状态导入入口，也不重构编辑器全局事件系统。真实阅读器页面隔离已经阻断本 P0 的作者事件泄漏；编辑器事件生命周期可在后续独立治理。
